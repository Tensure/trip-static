Current State

What TRIP Already Covers

The current application already has a strong base: dashboard, resource tracker, skill matrix, requirements, matching, and weight configuration. The enhancement should build on this rather than replace it.

Employee Intelligence

Employees, skills, deployability, ratings, bench status, availability, resume upload, feedback, and project data.

Matching Foundation

Weighted scoring exists and can be strengthened into role-wise matching with saved shortlists and clear reasons.

Spreadsheet Gap

Staffing plan, region-wise bench, project history, and demand-to-allocation lifecycle still live mostly in sheets and manual discussions.

Operating Flow

End-to-End Staffing Lifecycle

Every enhancement should support this flow. Once data enters TRIP, it should move forward naturally instead of being re-entered across spreadsheets, Slack threads, and employee profiles.

1

Client Demand

Manager captures client/project requirement and expected role demand.

2

Role Lines

Requirement is split into roles, counts, duration, skills, and status.

3

Matching

System ranks best-fit people per role with explainable scoring.

4

Shortlist

Managers save candidates, discuss, compare, and confirm selections.

5

Staffing Plan

Confirmed resource moves into active/upcoming staffing plan.

6

Project History

Allocation, feedback, role, and rolloff get written to employee profile.

7

Intelligence

Future matching improves through verified skills, feedback, and history.

Connected Data

One Entry, Reflected Across the System

The enhancement should avoid duplicate updates. When a manager confirms a staffing decision or updates project feedback, the same trusted record should appear in all relevant areas.

Requirement to Staffing

Once candidates are selected for a role line, the system should create or update staffing plan records and preserve the original requirement history.

Staffing to Employee Profile

Confirmed allocation should update employee current project, allocation percentage, bench status, availability date, and project assignment history.

Feedback to Intelligence

Project feedback entered from staffing or requirement pages should appear on the employee profile and improve future matching context.

Access Model

Simple Role-Based Access for MVP

Keep the first enhancement simple: three roles are enough. More granular roles can be added later only if needed.

Superadmin

  • Adds allowed manager users
  • Controls master entries and settings
  • Can view and manage all records
  • Owns system-level configuration

Manager

  • Uses the main TRIP application
  • Manages team resources and staffing
  • Approves team member submitted updates
  • Creates requirements and runs matching

Team Member

  • Uses self-service form/page only
  • Can update own data
  • Cannot access main manager app
  • Submissions require manager approval
Login and access rules

Main TRIP AppOnly Superadmin-created manager users can login. Google Workspace authentication should be used, but domain email alone is not enough for app access.
Self-Service FormAny valid @tensure.io team member can access the self-service page, but only to view or submit their own data.
Approval GateEmployee submitted changes remain pending until the mapped manager approves, edits, or rejects them.
Future RolesLeadership, HR, staffing admin, or viewer roles can be added later without changing the core model.

Capability Masters

  • Technical skill taxonomy
  • Domain and industry taxonomy

  • AI readiness categories

Governance

Audit Trail for Every Important Change

Since staffing and capability data affects decisions, the system should always show who changed what and when.

Audit Events

  • Employee profile changed
  • Skill approved or rejected
  • Requirement status changed
  • Candidate shortlisted or staffed
  • Staffing plan allocation updated
  • Master entry changed

Audit Fields

  • User and role
  • Timestamp
  • Old value and new value
  • Source: manual, self-service, resume extraction, matching engine
  • Approval state and approver comments

Cost-Aware Intelligence

AI and Semantic Scan Scope

The proposal should avoid making paid AI a dependency for the first enhancement. Use structured rules and lightweight extraction first. Add deeper AI only where business value is clear.

Phase 1 Enhancement

Low-cost intelligence using code and structured data.

Recommended
  • Algorithmic role-wise matching
  • Configurable scoring weights
  • Explainable match reasons
  • Basic resume text extraction
  • Known skill and certification detection from master lists
  • Manager approval before extracted data becomes official

Phase 2 Optional

Deeper semantic understanding if leadership approves AI cost.

Optional
  • Semantic resume scan using AI or embeddings
  • Requirement text parsing
  • AI-generated employee summaries
  • AI-generated match explanations
  • Natural language staffing search
  • Bench optimization suggestions
Semantic scan recommendation

Semantic scan is valuable, but should be positioned as a phased capability. The first enhancement can extract resume text and match against known master skills without heavy AI cost. Later, semantic scan can understand related concepts, infer domains, and detect skills that are not written exactly as master skill names.

Delivery Plan

Recommended Phased Roadmap

A phased roadmap keeps the enhancement practical and avoids trying to build every future intelligence idea at once.

Phase 1

Access, Masters, Self-Service

Google login, simple roles, manager allowlist, employee self-service form, approval queue, master entries.

Phase 2

Staffing Plan and Resource Views

Spreadsheet replacement, region-wise bench, internal staffing, active/upcoming/historical allocation tracking.

Phase 3

Requirements 2.0 and Matching

Multi-role requirements, discussion threads, role-wise matching, shortlist save, confirm staffing flow.

Phase 4

Search, Audit, Optional AI

Global search, complete audit trail, basic resume extraction, optional semantic scan and AI-assisted features.
